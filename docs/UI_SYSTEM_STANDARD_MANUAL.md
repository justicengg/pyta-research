# UI 系统标准手册

## 适用范围

本文档用于约束后续所有同学在持续扩展这套 UI 系统时的行为边界。

当前 UI 系统的视觉与交互基线文件为：
- `/Users/sikaijiang/Desktop/pyta-research/docs/multi_agent_sandbox_fusion_direction.html`

后续所有页面、组件、功能扩展，必须以这份基线为准，不允许绕开本手册随意新建视觉规则。

---

## 核心原则

后续扩展时，必须满足以下三条：

1. 不要破坏原有的字体
2. 不要破坏原有的颜色
3. 遵循原有的标准，不私自发明新的视觉系统

这三条不是建议，是约束。

---

## 1. 字体标准

## 1.1 唯一字体栈

当前系统唯一合法字体栈为：

```css
--sans: "Inter", "Noto Sans SC", -apple-system, BlinkMacSystemFont, sans-serif;
```

含义：
- 英文、数字、界面系统文字以 `Inter` 为主
- 中文以 `Noto Sans SC` 为主
- 系统字体仅作为兜底，不作为主动设计选择

## 1.2 禁止事项

后续扩展中，禁止：
- 私自改成其他主字体
- 在局部组件里单独使用另一套字体体系
- 因为“更酷”而引入新的 display font
- 混用 serif、monospace 作为 UI 主字体
- 使用系统默认字体替代现有字体栈

## 1.3 字号层级

必须沿用当前既有字号 token：

```css
--fs-xs:  11px;
--fs-sm:  14px;
--fs-md:  18px;
--fs-lg:  22px;
--fs-xl:  27px;
--fs-2xl: 34px;
```

规则：
- 优先直接使用现有 token
- 不要随意发明 `15px`、`17px`、`19px` 这种游离字号
- 如果真的需要新增字号，必须先评估是否能用现有层级解决
- 没有充分理由，不允许扩展字号体系

## 1.4 行高与字距

必须优先使用现有 token：

```css
--lh-tight:   1.25;
--lh-base:    1.5;
--lh-relaxed: 1.65;

--ls-tight:  -0.03em;
--ls-normal:  0em;
--ls-wide:    0.06em;
--ls-wider:   0.12em;
```

规则：
- 标题沿用紧字距
- 正文与说明文字优先用 `base` / `relaxed`
- 不要在局部随意写奇怪的字距和行高

## 2. 颜色标准

## 2.1 原则

颜色系统已经被定义为一套完整的 token 体系。
后续扩展时，必须复用，不允许随意换色。

## 2.2 亮色模式基线颜色

必须沿用以下语义色：

```css
--bg:               hsl(40, 18%, 95%);
--surface:          hsl(0, 0%, 100%);
--surface-subtle:   hsl(40, 14%, 98%);
--surface-overlay:  hsl(40, 10%, 96%);
--border:           hsl(35, 12%, 87%);
--border-subtle:    hsl(35, 10%, 92%);

--text-1:           hsl(25, 14%, 10%);
--text-2:           hsl(25,  8%, 36%);
--text-3:           hsl(25,  6%, 56%);
```

## 2.3 Accent 颜色

必须沿用现有 accent palette：

```css
--accent:           hsl(161, 72%, 28%);
--accent-fg:        hsl(161, 72%, 22%);
--lime:             hsl(72, 76%, 58%);
--lime-glow:        hsla(72, 76%, 58%, 0.15);
--blue:             hsl(218, 55%, 52%);
--orange:           hsl(24, 65%, 50%);
--purple:           hsl(258, 44%, 54%);
```

## 2.4 Agent 专属色

多 Agent 视觉必须复用现有轻 tint，不允许每个 Agent 自己一套饱和色。

```css
--tint-sandbox:     hsl(161, 38%, 96%);
--tint-macro:       hsl(218, 38%, 96%);
--tint-sentiment:   hsl(24,  45%, 96%);
--tint-user:        hsl(258, 32%, 96%);
--tint-decision:    hsl(145, 32%, 96%);
```

## 2.5 暗色模式

暗色模式已经定义完成，必须继续走同一套 token，不允许单独重新设计一个“第二套 UI”。

规则：
- 改组件时必须同时检查 light / dark / auto 三种模式
- 新增组件时必须接入同一套 token
- 禁止只做亮色、不管暗色

## 2.6 禁止事项

后续扩展中，禁止：
- 随意引入新主色
- 把系统改成高饱和度、强对比度风格
- 局部组件使用与全局不一致的背景色
- 随意写十六进制颜色绕开 token
- 为了“醒目”大量加纯黑、纯白、纯红

规则：
- 能用 token，就不要写裸色值
- 新增颜色前，先判断能否复用既有语义色
- 新增颜色必须具备系统级理由，而不是局部偏好

---

## 3. 间距标准

## 3.1 唯一间距网格

必须使用当前 8px grid：

```css
--sp-1:  4px;
--sp-2:  8px;
--sp-3:  12px;
--sp-4:  16px;
--sp-5:  20px;
--sp-6:  24px;
--sp-8:  32px;
--sp-10: 40px;
--sp-12: 48px;
```

规则：
- 所有 padding、gap、margin 优先从这套值里选
- 不要零散出现 `13px`、`18px`、`22px` 这种无系统依据的值
- 新组件必须先贴合网格，再谈局部微调

---

## 4. 圆角标准

必须复用现有圆角 token：

```css
--r-sm:   8px;
--r-md:   12px;
--r-lg:   16px;
--r-xl:   24px;
--r-2xl:  32px;
--r-full: 9999px;
```

规则：
- 小按钮 / 小条目：`sm` / `md`
- 卡片 / 面板：`lg` / `xl`
- 主容器：`2xl`
- pill / chip / tag：`full`

禁止：
- 同一页面里混用很多不相关圆角值
- 为了“特别”写出 20px、26px、30px 这类随机圆角

---

## 5. 阴影标准

必须优先使用：

```css
--shadow-sm
--shadow-md
--shadow-lg
--shadow-xl
```

规则：
- 不同层级通过既有 shadow token 区分
- 不要自己手写一堆新的 shadow 值
- 阴影要克制，服务层级，不服务炫技

禁止：
- 超重模糊阴影
- 彩色阴影泛滥
- 同一页面里出现不统一的投影风格

---

## 6. 布局标准

## 6.1 总体结构不变

当前系统的基线结构为：
- 左侧信息层边栏
- 中间研究画布
- 右侧决策架边栏
- 底部统一控制台

后续扩展时，不要轻易改变这个大结构。

## 6.2 边栏行为必须保留

左侧和右侧边栏的核心行为是：
- 可以收起
- 收起后变成一个小点
- 点击小点可重新展开
- 收起后中间画布自动扩展
- 底部对话框与画布同步自适应

这是系统级交互，不允许在新页面里丢掉。

## 6.3 中间画布优先

所有新功能都要先思考：
- 它应该进入画布
- 还是应该进入边栏
- 还是应该进入底部控制台

不要回退成传统 SaaS 的“新页面 + 新 tab + 新后台卡片区”。

---

## 7. 组件扩展标准

## 7.1 新组件必须先复用已有语义层

新增组件时，优先复用：
- `surface`
- `surface-subtle`
- `surface-overlay`
- `text-1/2/3`
- `border` / `border-subtle`
- `shadow-*`
- `r-*`
- `sp-*`

而不是自己重造一套局部样式。

## 7.2 新组件要保持角色一致

例如：
- 边栏组件应该继续像“rail”
- 画布对象应该继续像“node”
- 右栏内容应该继续像“收敛结果”
- 底部输入应该继续像“控制台”

不能新增一个组件，就把整个产品气质带回传统管理后台。

## 7.3 扩展顺序

每次扩展 UI 时，按这个顺序检查：
1. 是否复用了现有 token
2. 是否破坏了现有字体体系
3. 是否破坏了现有颜色体系
4. 是否破坏了中间画布优先的结构
5. 是否让第一层按钮变多了

只有都通过，才能合入。

---

## 8. 允许扩展的范围

可以扩展，但必须在原有标准内：
- 新节点类型
- 新 Agent 类型
- 新状态标签
- 新右栏结果模块
- 新左栏来源模块
- 新 `+` 菜单项
- 新模式下的附加信息层

扩展方式必须是“在原系统里长出来”，而不是“另起一套视觉语言”。

---

## 9. 不允许的扩展方式

后续同学严禁：
- 擅自改主字体
- 擅自改主色系
- 用新的组件库默认样式覆盖现有风格
- 用内联裸色值破坏 token 体系
- 用随机字号破坏排版秩序
- 把功能做成一堆厚重卡片
- 把画布产品做回传统后台
- 为了“显眼”大量增加按钮和操作条

---

## 10. 推荐协作方式

每次准备做 UI 扩展前，建议先回答这 5 个问题：

1. 这个功能属于左边栏、画布、右边栏，还是底部控制台？
2. 这个组件能不能用现有 token 直接搭出来？
3. 这个改动会不会破坏现有字体和颜色？
4. 这个改动会不会增加第一层视觉噪音？
5. 这个改动是否保持了“研究画布”而不是“传统 SaaS 后台”的产品气质？

只要有一项答不上来，就不应该直接开做。

---

## 11. 最终约束

后续所有人扩展这套 UI 系统时，必须记住：

- 字体是系统资产，不能随意动
- 颜色是系统资产，不能随意动
- token 是标准，不是参考
- 结构是研究画布，不是后台管理页
- 扩展必须沿着现有系统生长，而不是重新发明一套新的系统

这份手册的目的不是限制创造力，而是保证持续扩展时，产品不会被一点点做散、做乱、做回普通 SaaS。
