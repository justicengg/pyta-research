## 一、总体目标

实现一个不是传统 SaaS 后台，而是“Canvas-native research OS”的界面。

界面结构：
- 左侧：信息层边栏
- 中间：研究画布 / 多 Agent 沙盘推演舞台
- 右侧：决策架边栏
- 底部：统一控制台式对话框

要求：
- 页面简洁
- 少按钮
- 功能尽量收进第二层
- 画布是核心
- 左右边栏可以收起成小点
- 画布和底部对话框具备自适应联动

## 二、必须实现的页面结构

### 1. 左侧边栏：信息层

需要包含：
- 标题区
- 搜索框
- 安全设置区
- 已接入来源区
- 实时事件流区

必须支持：
- 点击收起按钮后收起
- 收起后变成一个小点
- 点击小点后展开

安全设置区需要强调：
- API Key
- Token
- 私有 Endpoint
等敏感字段应在这里管理，不通过底部对话框直接暴露给模型层。

### 2. 右侧边栏：决策架

需要包含：
- 标题区
- 搜索框
- 当前摘要
- 风险提示
- 动作队列
- 决策卡 / 执行建议

必须支持：
- 点击收起按钮后收起
- 收起后变成一个小点
- 点击小点后展开

注意：
右栏只负责收纳推演收敛后的结果，不承载完整推演过程。

### 3. 中间区域：研究画布

中间区域必须是统一画布，而不是多个互相割裂的大卡片容器。

需要包含：
- 核心场景节点
- 多个 Agent 节点
- 事件节点
- 中间判断节点
- 宏观修正节点
- 当前活跃步骤节点
- 决策草稿节点
- 图例

视觉方向：
- 中心舞台
- 多 Agent 围绕中心分布
- 时间流以细线、脉冲、高亮、淡化表达
- 不能做成硬泳道流程图
- 不能做成传统后台卡片墙

### 4. 底部控制台（Command Console）

底部区域不是普通聊天框，而是系统控制台。

必须包含：
- 一个大输入区
- 输入区下方左侧的 `+` 菜单
- 模式切换选择器

模式包括：
- 极简模式（默认）
- 专家模式

模式选择样式：
- 轻量文本式
- 不要厚重 button
- 使用轻量下拉箭头

`+` 菜单包含：
- 连接来源
- 加载技能
- 加载 Agent
- 场景设置
- 安全配置入口

注意：
- `+` 和模式选择都在对话框内部
- 工具区位于输入区下方左侧
- 不要使用粗暴横线切割布局

## 三、必须实现的交互

### 1. 左右边栏收起/展开联动

要求：
- 左边栏收起后，中间画布自动变宽
- 右边栏收起后，中间画布自动变宽
- 两侧都收起后，中间画布拿到最大空间
- 底部对话框宽度和位置必须随中间画布同步自适应
- 保持整体视觉对称

这是硬要求，不能只把边栏视觉隐藏而不联动布局。

### 2. 节点拖拽

中间画布中的以下对象必须可自由拖拽：
- 核心场景节点
- 各个 Agent 节点
- 事件节点
- 判断节点
- 决策草稿节点
- 图例

要求：
- 拖拽时手感稳定
- 节点不会瞬移
- 不要破坏画布整体结构

### 3. 工具条拖拽

画布中的工具条也要支持拖拽。
工具条内容包括：
- 添加事件
- 添加 Agent
- 添加场景
- 添加观点
- 连接来源
- 运行推演

### 4. 画布平移与缩放

中间画布需要支持基础画布交互：
- 鼠标滚轮 / 触控板缩放
- 拖动画布空白区域平移

目标体验：
- 在 Mac 触控板上自然
- 不依赖笨重的坐标箭头
- 用户可在大画布中左右移动和放大缩小

如果实现难度可控，也可以保留一个很轻的方向/缩放辅助控件，但它不能喧宾夺主。

### 5. 对话框行为

底部对话框要求：
- 输入区始终在底部
- 宽度与画布联动
- 左右边栏变化时保持对称
- 不要固定死在某一边

## 四、视觉与样式要求

### 1. 风格关键词

- 简洁
- 克制
- 有呼吸感
- 研究工作台
- 不像传统 SaaS
- 不像复杂运维后台
- 不像卡片堆叠站

### 2. 视觉约束

- 尽量减少显式按钮
- 默认第一层信息密度不能太高
- 大部分功能进第二层
- Agent 节点不要太厚重
- 中心舞台要有明确存在感
- 时间流痕迹要轻，不要变成流程图

### 3. 动效建议

可以加轻量动效，但不要过度：
- 边栏收起展开过渡
- 节点悬浮轻反馈
- 当前活跃步骤微高亮
- 时间流轻脉冲

避免：
- 夸张弹跳
- 过多浮层
- 炫技动画

## 五、技术实现建议

### 1. 状态建议

建议至少拆分以下状态：
- 左边栏展开/收起
- 右边栏展开/收起
- 当前模式（极简 / 专家）
- `+` 菜单打开/关闭
- 画布缩放比例
- 画布平移偏移量
- 当前拖拽对象

### 2. 组件建议

建议拆成：
- `InformationRail`
- `DecisionRail`
- `ResearchCanvas`
- `CanvasNode`
- `AgentNode`
- `CommandConsole`
- `ModeSelector`
- `PlusMenu`
- `CanvasToolbar`

### 3. 画布建议

如果使用 React，可考虑：
- 先手写 DOM + transform 实现基础拖拽/缩放
- 或使用轻量画布交互库，但不要引入过重依赖

重点不是图形引擎，而是：
- 手感稳定
- 布局清晰
- 后续易扩展成多 Agent 真正产品

- 把所有功能都放到第一层

## 八、最终判断标准

实现完成后，应该满足以下感受：
- 用户进入页面后，感觉像进入一个研究操作画布
- 左边输入，右边收敛，中间思考
- 对话框是控制台，不是普通聊天框
- 画布可组织、可移动、可放大缩小
- 多 Agent 推演是可感知的，而不是被藏在后台逻辑里
- 整体简洁、舒服、有秩序感
